## 1. Executive Summary

This report presents a static audit of the ZedApply codebase to identify bugs, architectural issues, security vulnerabilities, contract drift, and performance bottlenecks. Below is the ranked list of the top 10 findings.

### Top 10 Ranked Issues

* **P0 — CI Blocker in Migration Guard** — Pre-existing duplicate prefixes `063` and `074` in `master` cause the `migration-guard.yml` prefix-collision check to fail on **any new PR** that modifies migrations, blocking all automated migration dry-runs.
  * *Consequence:* CI validation of database schemas is completely broken for future changes.
  * *Location:* [.github/workflows/migration-guard.yml:15-22](file:///c:/Users/User/Documents/Claude/Projects/ZedApply/.github/workflows/migration-guard.yml#L15-L22)
* **P0 — Missing RLS on Aptitude Question Bank** — Row Level Security (RLS) is not enabled on `aptitude_question_bank`. Anyone querying Supabase via the public `anon` key can read all reasoning questions and answers.
  * *Consequence:* High fraud risk; users can retrieve correct test values directly from the database.
  * *Location:* [infra/supabase/migrations/057_interview_prep.sql:39-47](file:///c:/Users/User/Documents/Claude/Projects/ZedApply/infra/supabase/migrations/057_interview_prep.sql#L39-L47)
* **P0 — Missing RLS on Applications Table** — Row Level Security is not enabled on `applications`. Any user can read or modify application entries belonging to other users.
  * *Consequence:* Direct breach of user privacy and data integrity.
  * *Location:* [infra/supabase/migrations/003_frontend_overhaul_and_tiers.sql:29-38](file:///c:/Users/User/Documents/Claude/Projects/ZedApply/infra/supabase/migrations/003_frontend_overhaul_and_tiers.sql#L29-L38)
* **P1 — Undone search_path Hardening on `set_welcome_bonus()`** — Redefining `set_welcome_bonus()` in a later migration without declaring `SET search_path` resets its search path, completely undoing the security invoker hardening applied earlier.
  * *Consequence:* Vulnerable to search path hijacking/injection attacks as a `SECURITY DEFINER` function.
  * *Location:* [infra/supabase/migrations/096_free_tier_promo_one_month.sql:6-22](file:///c:/Users/User/Documents/Claude/Projects/ZedApply/infra/supabase/migrations/096_free_tier_promo_one_month.sql#L6-L22)
* **P1 — Ineffective DB Security via Global `service_role` Bypass** — The backend client uses the cached `service_role` key globally, which bypasses all RLS checks. Isolating user data depends solely on explicit filter clauses in code.
  * *Consequence:* A single missing `.eq("user_id", user_id)` filter in FastAPI endpoints leaks data across tenants.
  * *Location:* [apps/backend/app/core/deps.py:22](file:///c:/Users/User/Documents/Claude/Projects/ZedApply/apps/backend/app/core/deps.py#L22)
* **P1 — N+1 Performance Landmine in Nightly Batch Matching** — The nightly cron loop processes matching for active users sequentially, triggering multiple database requests per user.
  * *Consequence:* High latency, database connection exhaustion, and cron timeouts under moderate user load.
  * *Location:* [apps/backend/app/services/batch_matching.py:166-173](file:///c:/Users/User/Documents/Claude/Projects/ZedApply/apps/backend/app/services/batch_matching.py#L166-L173)
* **P1 — N+1 Performance Landmine in Daily Digest Cron** — Matches, interviews, and delivery notifications are resolved sequentially in loops inside `run_email_daily_digest()` and `run_whatsapp_daily_digest()`.
  * *Consequence:* Cron execution halts, causing timeouts and partial delivery of digests.
  * *Location:* [apps/backend/app/services/daily_digest.py:276-311](file:///c:/Users/User/Documents/Claude/Projects/ZedApply/apps/backend/app/services/daily_digest.py#L276-L311) and [daily_digest.py:324-366](file:///c:/Users/User/Documents/Claude/Projects/ZedApply/apps/backend/app/services/daily_digest.py#L324-L366)
* **P1 — Admin Job Edit 422 Crash via Forbidden Extra Fields** — The backend schema `AdminJobUpdate` uses `extra="forbid"` but is missing fields that are returned by the job detail GET endpoint (e.g. `whatsapp_message_id`, `ocr_source_text`).

* **P2 — Dead Code: In-App User Notifications Router** — The user-facing notification endpoints are completely defined in the backend but have no frontend caller, and the associated table is never pruned.
  * *Consequence:* The database accumulates unread notifications indefinitely without ever rendering them.

## 5. Endpoint Security

### Missing Role Check in Admin Endpoints
* Endpoints defined in `apps/backend/app/api/v1/admin_ingest.py` (e.g. `/trigger-daily-digest-email`) rely on `_require_cron_auth` which only verifies the `ADMIN_API_KEY` or `INGEST_API_KEY` headers. 
* While standard admin HTTP routes properly enforce `Depends(require_admin)` (which checks the user's role column in the database), the cron paths rely purely on static token validation. If the admin/ingest key is leaked, attackers gain unrestricted ability to trigger batch runs, send emails, and modify configuration.

### Sensitive Data Returns
* `GET /admin/bwana/conversations` ([admin_bwana.py:97](file:///c:/Users/User/Documents/Claude/Projects/ZedApply/apps/backend/app/api/v1/admin_bwana.py#L97)) returns previews of raw chatbot conversations from `ai_cache` which contain user phone numbers, names, and CV content. This endpoint returns raw user PII to the admin console without encryption or redacting.
